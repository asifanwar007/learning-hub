⚡ Redis Caching

High-performance in-memory data store for caching, sessions, and real-time analytics

📋 Executive Summary

What: Redis is an in-memory data structure store used for caching, session management, real-time analytics, and pub/sub messaging.

Performance: Sub-millisecond latency. Can handle 100K+ operations per second on a single instance.

When to use: Need fast reads, caching database queries, session storage, leaderboards, real-time counters

Common use cases: Cache layer, session store, rate limiting, job queues, real-time analytics

Data loss risk: In-memory only (unless persistence enabled). Data lost on restart unless saved to disk.

Overview & Core Concepts

Redis stores data in-memory with optional persistence to disk. It's much faster than databases (network + disk I/O) but has less data (only what fits in RAM).

Why Redis is Fast

Operation PostgreSQL Redis Speed Gain
Get user by ID 5-10ms 0.1-0.5ms 50-100x faster
Cache miss + DB hit 10-15ms 10-15ms Same (with miss)
100K operations 1-2 seconds 100-200ms 10-15x faster

How Junior Developers Use Redis

Install & Run Locally

# Using Docker
docker run -d -p 6379:6379 redis:latest

# Or using Homebrew (macOS)
brew install redis
redis-server

Basic Cache Operations

// Java with Jedis
import redis.clients.jedis.Jedis;

Jedis redis = new Jedis("localhost", 6379);

// Set a value (string)
redis.set("user:123:name", "John Doe");
redis.expire("user:123:name", 3600);  // 1 hour expiry

// Get a value
String name = redis.get("user:123:name");  // "John Doe"

// Delete
redis.del("user:123:name");

// Key expiration is CRITICAL for cache
redis.setex("session:abc", 1800, "session_data");  // Expires in 30 min

Common Caching Pattern

// Cache-Aside Pattern (most common)
public User getUser(Long userId) {
    // Check cache first
    String cached = redis.get("user:" + userId);
    if (cached != null) {
        return deserialize(cached);  // Fast path
    }

    // Cache miss - hit database
    User user = database.findById(userId);

    // Store in cache for next time
    redis.setex("user:" + userId, 3600, serialize(user));

    return user;
}
⚠️ Beginner Mistakes
  • 🚨 Forgetting to set expiration (cache grows forever, OOM)
  • 🚨 Caching user passwords (security risk)
  • 🚨 Complex nested objects without serialization
  • 🚨 No error handling for Redis failures
  • 🚨 Assuming Redis is persistent (it's not by default)

How Senior Developers Use Redis

Distributed Cache with Fallback

public User getUser(Long userId) {
    try {
        String cached = redis.get("user:" + userId);
        if (cached != null) {
            metrics.increment("cache.hit");
            return deserialize(cached);
        }
    } catch (RedisConnectionException e) {
        logger.warn("Redis unavailable, falling back to DB");
        // Continue to database
    }

    metrics.increment("cache.miss");
    User user = database.findById(userId);

    // Cache with short TTL to avoid stale data
    try {
        redis.setex("user:" + userId, 600, serialize(user));  // 10 min
    } catch (RedisConnectionException e) {
        logger.warn("Could not cache, continuing");
    }

    return user;
}
✅ Production Patterns
  • Always handle connection failures gracefully
  • Use appropriate TTLs (not too long, not forever)
  • Monitor cache hit rate (should be >80%)
  • Set up replication for high availability
  • Use sentinel for automatic failover
  • Implement cache invalidation strategy
  • Monitor memory usage (set max-memory policy)

Cache Invalidation (Most Important)

// When user updates their data
public void updateUser(Long userId, User newData) {
    // Update database first
    database.save(newData);

    // Invalidate cache immediately (write-through)
    redis.del("user:" + userId);

    // Publish invalidation event for other services
    redis.publish("user:updated", userId.toString());
}

How Architects Think About Redis

When to Use Redis

Scenario Redis? Decision
Read-heavy workload, fast data ✅ Yes. Cache hot data
Session storage ✅ Perfect. Fast, expires automatically
Real-time leaderboards/counters ✅ Yes. INCR is atomic and fast
Rate limiting ✅ Yes. INCR with TTL
Persistent primary data ❌ No. Use database
Large datasets > available RAM ❌ No. Spill to disk ruins performance

Cost Implications

  • Infrastructure: 2-4x cost of database (need high-RAM instances)
  • Operational: Monitor memory, handle evictions, manage persistence
  • Reliability: Must run replicas for HA (doubles cost)
  • Latency benefit: Worth it for high-traffic systems

Benefits & Pros

  • ✅ Ultra-fast: Sub-millisecond latency
  • ✅ Reduces database load: Cache popular queries
  • ✅ Simple API: Easy to use key-value store
  • ✅ Flexible data structures: Strings, Lists, Sets, Hashes, Sorted Sets
  • ✅ Atomic operations: INCR, LPUSH are atomic

Pitfalls & Cons

❌ Critical: Data Loss Risk

Redis keeps data in RAM. Server crashes = data lost. Solution: Enable AOF (append-only file) or RDB persistence.

⚠️ Cache Invalidation Complexity

Phil Karlton: "There are only two hard things in computing: cache invalidation and naming things."

If you don't invalidate properly, users see stale data.

⚠️ Memory Pressure

Cache grows. If not careful, Redis uses all RAM and starts evicting data (hurts performance).

Best Practices

1. Always Set Expiration (TTL)

// Good
redis.setex("key", 3600, "value");  // Expires in 1 hour

// Bad
redis.set("key", "value");  // Never expires!

2. Monitor Cache Hit Rate

// Track hits and misses
if (redis.exists(key)) {
    metrics.increment("cache.hit");
} else {
    metrics.increment("cache.miss");
}

// Target: >80% hit rate

3. Handle Failures Gracefully

try {
    return redis.get(key);
} catch (RedisConnectionException e) {
    logger.warn("Redis failed, using fallback");
    return database.query();  // Fallback
}

4. Use Connection Pooling

// Good: Pool reuses connections
JedisPool pool = new JedisPool(new JedisPoolConfig(), "localhost");
Jedis jedis = pool.getResource();
try {
    jedis.get("key");
} finally {
    jedis.close();
}

Real-World Examples

Session Caching

public void saveSession(String sessionId, SessionData data) {
    redis.setex("session:" + sessionId, 1800, serialize(data));
}

public SessionData getSession(String sessionId) {
    String data = redis.get("session:" + sessionId);
    if (data == null) throw new SessionExpiredException();
    return deserialize(data);
}

Rate Limiting

public boolean isAllowed(String userId) {
    String key = "rate:" + userId;
    Long count = redis.incr(key);

    if (count == 1) {
        redis.expire(key, 60);  // 1 minute window
    }

    return count <= 100;  // Allow 100 requests per minute
}

Key Takeaway: Redis is incredibly fast but requires careful cache invalidation and memory management. Perfect for caching and sessions. Not for persistent primary data.
