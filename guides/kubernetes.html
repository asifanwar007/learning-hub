🐳 Kubernetes

Container orchestration platform for deploying, scaling, and managing containerized applications

📋 Executive Summary

What: Kubernetes (K8s) is an open-source container orchestration platform that automates deployment, scaling, and management of containerized workloads.

Scale: Powers production deployments at Google, Amazon, Microsoft, and manages millions of containers daily

When to use: Multi-container applications, microservices, need for auto-scaling, high availability requirements

Core concepts: Pods, Services, Deployments, ConfigMaps, Secrets, Ingress

Learning time: 2-4 weeks for production proficiency

Overview & Core Concepts

Kubernetes is a declarative platform for orchestrating containerized applications. Instead of manually managing Docker containers across servers, you declare what you want, and Kubernetes makes it happen.

Core Architecture

Your Application (Microservices)
        ↓
Docker Containers
        ↓
Kubernetes Cluster
├─ Control Plane (Master)
│  ├─ API Server
│  ├─ Scheduler
│  └─ Controller Manager
└─ Worker Nodes
   ├─ kubelet
   ├─ Container Runtime
   └─ kube-proxy

Key Concepts

Concept What It Does Real Example
Pod Smallest deployable unit. One or more containers sharing network. Your API server running in a container
Service Network abstraction. Internal DNS + load balancer for Pods. api-service.default.svc.cluster.local
Deployment Declarative updates. Says "run 3 replicas of my app" 3 copies of API server, auto-heals if one dies
ConfigMap Configuration as a resource (non-secret) Database host, API timeout values
Secret Encrypted configuration for sensitive data Database passwords, API keys
Ingress HTTP(S) routing into cluster Route api.example.com → API service

How Junior Developers Use Kubernetes

Start with the basics. Don't worry about advanced networking yet.

Your First Deployment

# 1. Write a deployment manifest (deployment.yaml)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:1.0
        ports:
        - containerPort: 8080

# 2. Deploy to Kubernetes
kubectl apply -f deployment.yaml

# 3. Check status
kubectl get deployments
kubectl get pods
kubectl logs pod-name

# 4. Expose your app
kubectl expose deployment my-app --type=LoadBalancer --port=80 --target-port=8080
⚠️ Common Beginner Mistakes
  • 🚨 Using latest tag instead of specific versions (unpredictable)
  • 🚨 No resource requests/limits (pods can crash cluster)
  • 🚨 Storing secrets in ConfigMaps (security risk)
  • 🚨 Not setting up health checks (zombie pods)
  • 🚨 Running as root in containers (security issue)

Essential kubectl Commands

# View resources
kubectl get pods
kubectl get services
kubectl get deployments
kubectl describe pod pod-name

# Debug
kubectl logs pod-name

# Create/Update

# Delete
kubectl delete pod pod-name
kubectl delete deployment my-app

How Senior Developers Use Kubernetes

Focus on reliability, observability, and production patterns.

Production-Grade Deployment

# deployment.yaml with production patterns
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
  namespace: production
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: api-server
      version: v2
  template:
    metadata:
      labels:
        app: api-server
        version: v2
    spec:
      # Pod disruption budget for high availability
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - api-server
              topologyKey: kubernetes.io/hostname

      containers:
      - name: api-server
        image: registry.example.com/api-server:2.0.1
        imagePullPolicy: IfNotPresent

        # Resource management (critical!)
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"

        # Health checks
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10

        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5

        # Environment configuration
        env:
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: db-host
        - name: DB_PASS
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password

        # Graceful shutdown
        lifecycle:
          preStop:
            exec:
              command: ["/bin/sh", "-c", "sleep 15"]

      # Pod-level settings
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000

      # Pod disruption budget (PDB)
---
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-server-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: api-server
✅ Production Patterns
  • Always use specific image versions (not "latest")
  • Set resource requests/limits (prevents cluster collapse)
  • Implement liveness & readiness probes (automatic recovery)
  • Use PodDisruptionBudgets (maintain availability during updates)
  • Pod anti-affinity (spread across nodes)
  • SecurityContext (non-root users)
  • RollingUpdate strategy (zero-downtime deploys)

Debugging Production Issues

# Check pod events (most useful!)
kubectl describe pod problem-pod

# View recent logs with context
kubectl logs pod-name --tail=100 -f

# Check resource usage
kubectl top pods
kubectl top nodes

# Check node status
kubectl get nodes -o wide
kubectl describe node node-name

# Check for scheduling issues
kubectl get events --sort-by='.lastTimestamp'

# Debug networking
kubectl run debug --image=nicolaka/netshoot -it --rm -- bash
# Then: nslookup service-name, curl http://service-name:port, etc

How Architects Think About Kubernetes

When to Use Kubernetes

Scenario K8s? Why/Why Not
Single monolithic app ❌ Overkill. Use Docker + VMs or Docker Swarm
10+ microservices, high traffic ✅ Perfect fit. Auto-scaling, load balancing, resilience
Batch jobs, data processing ✅ Yes. Job/CronJob resources for scheduled work
Real-time games, low latency critical ⚠️ Maybe. K8s adds 50-100ms latency overhead
Startup with <3 engineers ❌ Don't. Too much operational burden
Enterprise with DevOps team ✅ Absolutely. ROI on tooling and automation

Cost Implications

  • Infrastructure cost: 25-40% higher than VMs (due to overhead, wasted resources)
  • Learning cost: 3-6 months for team to be production-ready
  • Operational cost: High initially (setup, monitoring, debugging), drops long-term
  • Time-to-value: 3-6 months before seeing productivity gains

Alternatives Matrix

Solution Complexity Auto-Scaling Cost Best For
Docker Swarm Low Basic Very Low Small teams, simple apps
Kubernetes High Excellent High Large scale, microservices
Serverless (Lambda) Low Automatic Per-execution Event-driven, spiky workloads
PaaS (Heroku) Very Low Built-in Very High Rapid prototyping

Benefits & Pros

1. Automatic Scaling

# Horizontal Pod Autoscaling (HPA)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

✅ Result: Automatically scales from 3 to 10 pods based on CPU usage. Zero intervention.

2. High Availability & Self-Healing

✅ Pod crashes? Kubernetes restarts it. Node dies? Pods move to healthy nodes. No manual intervention needed.

3. Rolling Updates (Zero Downtime)

✅ Deploy new version, Kubernetes gradually replaces old pods with new ones. Users never experience downtime.

4. Declarative Configuration

✅ Version control your infrastructure. Git becomes your source of truth.

5. Multi-Cloud & Portability

✅ Same manifests work on AWS EKS, Google GKE, Azure AKS, on-premises. Vendor lock-in reduced.

Pitfalls & Cons

❌ Critical Pitfall #1: Resource Limits Kill Your Pods

If you set memory limit too low, pods get OOMKilled (out of memory) randomly.

// 🚨 BAD - Memory limit too aggressive
resources:
  limits:
    memory: "128Mi"  # Too low for real app!

// ✅ GOOD - Set reasonable limits
resources:
  requests:
    memory: "256Mi"  # Minimum needed
  limits:
    memory: "512Mi"  # Can spike to this
❌ Critical Pitfall #2: No Resource Requests

Pods with no requests get scheduled anywhere, causing cascading failures.

⚠️ Pitfall #3: Complexity Explosion

Kubernetes is powerful but complex. Easy to over-engineer. Start simple.

⚠️ Pitfall #4: Debugging is Hard

Network issues, pod scheduling, image pull errors are cryptic. Need strong observability.

⚠️ Pitfall #5: Cost Surprises

If not managed, cloud bills can explode (idle pods, persistent volumes, load balancers).

Best Practices & Guardrails

1. Always Use Namespaces

kubectl create namespace production
kubectl apply -f deployment.yaml -n production

2. Resource Quotas Per Namespace

apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    requests.cpu: "10"
    requests.memory: "20Gi"
    limits.cpu: "20"
    limits.memory: "40Gi"

3. Network Policies (Firewall Rules)

# Only allow API to DB traffic
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-network-policy
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: api
    ports:
    - protocol: TCP
      port: 5432

4. Always Use RBAC

Don't give everyone full cluster access. Use Role-Based Access Control.

5. Monitor Everything

Implement Prometheus + Grafana for metrics, ELK for logs, Jaeger for traces.
