⚡ Circuit Breaker

Resilience pattern for preventing cascading failures in distributed systems

📋 Executive Summary

What: Circuit breaker is a design pattern that prevents your application from making requests to a failing service. Like an electrical circuit breaker: stops power flow when there's a problem.

Why: When a service is down/slow, repeated requests waste resources and delay failure detection. Circuit breaker detects failure and fast-fails instead of waiting.

Three states: CLOSED (requests allowed) → OPEN (requests blocked) → HALF_OPEN (testing recovery)

When to use: Any external service call (REST API, database, cache) that could fail

Key benefit: Prevents cascading failures. If payment service is down, you fail fast instead of hanging for 30 seconds.

Learning curve: Easy to understand, critical for production systems

Core Concepts

Three States Explained

State Behavior When It Happens
CLOSED Requests pass through normally. Failures counted. Service is healthy. This is the normal state.
OPEN Requests immediately fail (fast-fail). No calls to the service. Failure threshold reached (e.g., 5 failures in 10 seconds)
HALF_OPEN Limited requests allowed to test if service recovered. After timeout in OPEN state. Testing if service is back up.

State Transitions

CLOSED → OPEN
When: Failure rate exceeds threshold (e.g., 5 failures in 10 seconds)
Action: Start blocking requests

OPEN → HALF_OPEN
When: Timeout reached (e.g., 30 seconds after opening)
Action: Allow limited requests to test recovery

HALF_OPEN → CLOSED
When: Test request succeeds
Action: Circuit closes, requests flow normally again

HALF_OPEN → OPEN
When: Test request fails
Action: Circuit opens again, reset timeout counter

How Junior Developers Use Circuit Breaker

Basic Circuit Breaker Pattern

// Without Circuit Breaker (BAD 🚨)
public User getUser(String userId) {
    try {
        // If service is down, this waits 30 seconds
        return userService.callRemoteApi(userId);
    } catch (Exception e) {
        throw e;  // Caller has to wait for timeout
    }
}

// With Circuit Breaker (GOOD ✅)
CircuitBreaker breaker = new CircuitBreaker(
    failureThreshold: 5,  // Open after 5 failures
    timeout: 30000        // Try recovery after 30s
);

public User getUser(String userId) {
    // Immediately fails if circuit is OPEN (no timeout waiting)
    return breaker.execute(() -> userService.callRemoteApi(userId));
}

Using Resilience4j (Most Popular)

// Maven dependency
//

import io.github.resilience4j.circuitbreaker.CircuitBreaker;
import io.github.resilience4j.circuitbreaker.CircuitBreakerConfig;

// Configure circuit breaker
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50.0f)  // Open if 50% fail
    .slowCallRateThreshold(50.0f)  // Or if 50% are slow
    .slowCallDurationThreshold(Duration.ofSeconds(2))
    .minimumNumberOfCalls(10)  // Need 10 calls to evaluate
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .build();

CircuitBreaker breaker = CircuitBreaker.of("myBreaker", config);

// Use it
public User getUser(String userId) {
    return breaker.executeSupplier(() -> userService.getUser(userId));
}

With Fallback

public User getUser(String userId) {
    try {
        return breaker.executeSupplier(() -> userService.getUser(userId));
    } catch (CallNotPermittedException e) {
        // Circuit is OPEN - use fallback
        logger.warn("User service is down, returning cached data");
        return userCache.get(userId);  // Stale but available
    }
}
⚠️ Beginner Mistakes
  • 🚨 Not setting proper timeout (use 2-5 seconds, not 30)
  • 🚨 Failure threshold too low (you want some retries first)
  • 🚨 No fallback strategy (circuit opens but you have no Plan B)
  • 🚨 Applying circuit breaker to local calls (not needed)
  • 🚨 Forgetting to monitor circuit state (you won't know when it opens)

How Senior Developers Use Circuit Breaker

Production Configuration

// Configuration for payment service (critical path)
CircuitBreakerConfig paymentBreakerConfig = CircuitBreakerConfig.custom()
    .failureRateThreshold(25.0f)  // Aggressive: 25% failure threshold
    .slowCallRateThreshold(50.0f)  // Slow calls also count
    .slowCallDurationThreshold(Duration.ofSeconds(1))  // 1s is slow for payment
    .minimumNumberOfCalls(20)  // Need 20 calls to be sure
    .waitDurationInOpenState(Duration.ofSeconds(60))  // Wait 60s before retry
    .permittedNumberOfCallsInHalfOpenState(5)  // Test with 5 calls
    .automaticTransitionFromOpenToHalfOpenEnabled(true)  // Auto recovery test
    .build();

// Configuration for cache service (less critical)
CircuitBreakerConfig cacheBreakerConfig = CircuitBreakerConfig.custom()
    .failureRateThreshold(70.0f)  // Lenient: 70% failure threshold
    .minimumNumberOfCalls(5)
    .waitDurationInOpenState(Duration.ofSeconds(10))
    .build();

Combining with Retry & Timeout

// Retry first, then circuit breaker, then timeout
Retry retryPolicy = Retry.of("retry", RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .build());

Timeout timeout = Timeout.of(Duration.ofSeconds(2));

CircuitBreaker breaker = CircuitBreaker.of("api", CircuitBreakerConfig.custom()
    .failureRateThreshold(50)
    .build());

// Decorators applied in order: Timeout → Retry → CircuitBreaker
Supplier supplier = () -> externalApiCall();
Supplier decorated = Decorators.ofSupplier(supplier)
    .withTimeout(timeout)
    .withRetry(retryPolicy)
    .withCircuitBreaker(breaker)
    .decorate();

// Execution order:
// 1. Timeout: Max 2 seconds total
// 2. Retry: Up to 3 attempts with 500ms wait between
// 3. CircuitBreaker: Prevents cascading failure if threshold exceeded
String result = decorated.get();

Monitoring Circuit State

// Listen to state transitions
breaker.getEventPublisher()
    .onStateTransition(event -> logger.warn(
        "Circuit state changed: {} → {}",
        event.getStateTransition().getFromState(),
        event.getStateTransition().getToState()
    ))
    .onError(event -> metrics.increment("circuitbreaker.errors"))
    .onSuccess(event -> metrics.increment("circuitbreaker.success"))
    .onCallNotPermitted(event -> {
        logger.error("Request rejected - circuit is OPEN");
        metrics.increment("circuitbreaker.blocked");
        sendAlert("Payment API circuit breaker is OPEN");
    });

// Metrics to track
CircuitBreaker.Metrics metrics = breaker.getMetrics();
logger.info("Failure rate: {}%", metrics.getFailureRate());
logger.info("Slow call rate: {}%", metrics.getSlowCallRate());
logger.info("State: {}", breaker.getState());  // CLOSED, OPEN, HALF_OPEN

How Architects Think About Circuit Breaker

When to Use Circuit Breaker

Scenario Use Circuit Breaker? Why?
Call to external API (payment, SMS, email) ✅ Yes, always Prevents cascading failure, detects outages
Database queries (local DB) ❓ Sometimes Only if database is shared/remote, not for local queries
Microservice calls (REST/gRPC) ✅ Yes, always Other services can fail independently
Cache queries (Redis, Memcached) ✅ Yes Cache can be down, need fallback to DB
In-memory operations (local HashMap) ❌ No These don't fail externally
Critical path (payment, auth) ✅ Yes, with aggressive threshold Faster failure = better UX than hanging

Circuit Breaker vs Retry vs Timeout

Pattern Purpose Example When to Use
Retry Handle transient failures Network blip, 1s timeout Before circuit breaker
Timeout Bound wait time Stop waiting after 2s Prevent thread exhaustion
Circuit Breaker Prevent cascading failure Stop calling service when it's down After timeout/retry fail repeatedly

Cost Analysis

  • CPU: Negligible. Just counter checks and state transitions.
  • Memory: Minimal. Stores failure counts and timestamps.
  • Latency: ~1-2 microseconds per call (circuit breaker check overhead)
  • Benefit: Prevents 30-second hangs, improves UX, prevents resource exhaustion

Benefits & Pros

  • ✅ Prevents cascading failures: One service down doesn't take down the entire system
  • ✅ Fast failure: Detects failures immediately, doesn't wait for timeouts
  • ✅ Resource protection: Prevents thread pool exhaustion from hanging requests
  • ✅ Automatic recovery: HALF_OPEN state tests if service recovered
  • ✅ Monitorable: Can alert when circuit opens (service down)
  • ✅ Graceful degradation: With fallbacks, provides degraded service instead of complete failure

Pitfalls & Cons

❌ Distributed Circuit Breaker Problem

Circuit breaker state is local to each instance. Instance A might think service is down while Instance B doesn't. Inconsistent behavior. Solution: Use shared state (Redis) for large deployments.

⚠️ Cascading Failures with Wrong Config

If threshold is too aggressive, circuit opens prematurely. If too lenient, you get cascading failures anyway. Need to tune carefully for each service.

⚠️ Thundering Herd Problem

When circuit opens and all requests fast-fail, they might retry simultaneously when circuit closes. Use jitter in retry delays to prevent this.

⚠️ False Positives (HALF_OPEN Test Fails)

Circuit tests recovery with a single request. If that request fails, circuit opens again even though service might be up. Use multiple test requests.

⚠️ Fallback Complexity

Fallbacks need to be thought through. Returning cached data might return stale info. Using a default might hide real problems.

Best Practices

1. Always Use Timeout + Circuit Breaker

Never use circuit breaker alone. Timeout prevents hanging, circuit breaker prevents cascading.

// Good: Timeout then Circuit Breaker
Decorators.ofSupplier(supplier)
    .withTimeout(Duration.ofSeconds(2))
    .withCircuitBreaker(breaker)
    .decorate();

// Bad: Circuit Breaker without timeout
CircuitBreaker.of("api").executeSupplier(supplier);

2. Configure Different Thresholds for Different Services

// Payment API: Aggressive (fail fast on payment)
failureThreshold: 25%, waitDuration: 60s

// Logging API: Lenient (more tolerant)
failureThreshold: 70%, waitDuration: 10s

// Cache: Very lenient (can fail often)
failureThreshold: 90%, waitDuration: 5s

3. Implement Proper Fallback

public OrderResponse createOrder(Order order) {
    try {
        return breaker.executeSupplier(() -> paymentService.process(order));
    } catch (CallNotPermittedException e) {
        // Circuit is OPEN
        // Option 1: Return cached success (optimistic)
        return cache.getLastSuccess();

        // Option 2: Queue for later (pessimistic)
        // asyncQueue.add(order);
        // return OrderResponse.PENDING;

        // Option 3: Fail loudly
        // throw new PaymentServiceUnavailableException();
    }
}

4. Monitor Circuit State Changes

breaker.getEventPublisher()
    .onStateTransition(event -> {
        logger.error("Circuit {} changed: {} → {}",
            breaker.getName(),
            event.getStateTransition().getFromState(),
            event.getStateTransition().getToState());

        // Alert operations team
        if (event.getStateTransition().getToState() == State.OPEN) {
            alerting.sendAlert("Payment API is down!");
        }
    });

5. Use Jitter in Retry Delays

// Prevents thundering herd when service recovers
RetryConfig retryConfig = RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .intervalFunction(IntervalFunction.ofRandomized(
        500,           // Base: 500ms
        0.5            // Jitter: ±50%
    ))
    .build();

Guardrails

1. Never Skip Circuit Breaker for External Calls

Always wrap external API calls, microservice calls, and database calls with circuit breaker + timeout.

2. Alert When Circuit Opens

An open circuit means a critical service is down. Operations team needs to know immediately.

3. Use Multiple Test Requests in HALF_OPEN

Don't rely on a single test request. Use permittedNumberOfCallsInHalfOpenState to test with multiple requests.

4. Track Metrics: Failure Rate, Slow Call Rate, State

Monitor these continuously in production. A creeping failure rate indicates degradation.

5. Test Circuit Breaker Behavior

Write tests that simulate service failures and verify circuit breaker opens/closes correctly.

Real-World Example: Payment Service Integration

// Configuration for payment API (critical)
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(30.0f)  // Open after 30% failures
    .slowCallRateThreshold(50.0f)  // Open if 50% are slow
    .slowCallDurationThreshold(Duration.ofSeconds(1))
    .minimumNumberOfCalls(20)
    .waitDurationInOpenState(Duration.ofSeconds(60))
    .permittedNumberOfCallsInHalfOpenState(3)  // Test with 3 calls
    .build();

CircuitBreaker paymentBreaker = CircuitBreaker.of("payment", config);

// Monitoring
paymentBreaker.getEventPublisher()
    .onStateTransition(event -> {
        if (event.getStateTransition().getToState() == State.OPEN) {
            // CRITICAL: Payment service is down!
            alerting.sendSevereAlert("Payment API circuit opened");
            metrics.gauge("payment_api_down", 1);
        }
    });

// Usage with timeout and fallback
@Transactional
public PaymentResult processPayment(Order order) {
    try {
        return Decorators.ofSupplier(() ->
            paymentApiClient.charge(order.getAmount()))
            .withTimeout(Duration.ofSeconds(3))
            .withRetry(Retry.of("payment", RetryConfig.custom()
                .maxAttempts(2)
                .waitDuration(Duration.ofMillis(500))
                .build()))
            .withCircuitBreaker(paymentBreaker)
            .decorate()
            .get();
    } catch (CallNotPermittedException e) {
        // Payment API is down (circuit open)
        logger.error("Payment API unavailable, queuing for retry");
        paymentQueue.add(order);  // Queue for async retry
        return PaymentResult.PENDING;
    } catch (TimeoutException e) {
        logger.error("Payment API timeout");
        paymentQueue.add(order);
        return PaymentResult.PENDING;
    }
}

Key Takeaway: Circuit breaker prevents cascading failures by fast-failing when external services are down. Use with timeout and proper fallback strategy. Monitor state changes religiously in production.
