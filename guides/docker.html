🐳 Docker

Containerization platform for packaging applications with all dependencies

📋 Executive Summary

What: Docker packages your application code, libraries, and runtime into a container. "Works on my machine" becomes "works everywhere".

Why: Eliminates environment inconsistencies. Same container runs on laptop, CI/CD, staging, production.

Core concept: Image (blueprint) → Container (running instance)

When to use: Any application you want to ship consistently. Default choice for most apps.

Learning curve: 1-2 weeks to be productive, months to master

Core Concepts

Image vs Container

Image Container
Blueprint/Template Running instance
Immutable Mutable (logs, state changes)
Stored on disk Runs in memory
Created once, reused Created/destroyed frequently
Example: Docker image Example: Running process

How Junior Developers Use Docker

Build Your First Docker Image

# 1. Create Dockerfile
# Dockerfile
FROM python:3.11
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
EXPOSE 5000
CMD ["python", "app.py"]

# 2. Build image
docker build -t my-app:1.0 .

# 3. Run container
docker run -p 5000:5000 my-app:1.0

# 4. Container runs, app available at localhost:5000

# 5. Stop container
docker stop

Essential Docker Commands

# Build and manage images
docker build -t myapp:1.0 .
docker images
docker tag myapp:1.0 registry.com/myapp:1.0
docker push registry.com/myapp:1.0

# Run containers
docker run -p 8080:8080 myapp:1.0
docker run -it myapp:1.0 /bin/bash  # Interactive
docker run -d myapp:1.0  # Detached (background)

# Manage running containers
docker ps  # List running
docker ps -a  # List all
docker logs 
docker exec -it  /bin/bash
docker stop 
docker rm
⚠️ Beginner Mistakes
  • 🚨 Using "latest" tag (unpredictable)
  • 🚨 Not cleaning up images/containers (disk space)
  • 🚨 Running as root (security)
  • 🚨 Large images (slow push/pull)
  • 🚨 Forgetting .dockerignore (includes unnecessary files)

How Senior Developers Use Docker

Production-Grade Dockerfile

# Multi-stage build (reduces image size)
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:18-slim
WORKDIR /app
# Run as non-root user
RUN useradd -m -u 1000 appuser
COPY --from=builder --chown=appuser:appuser /app/dist ./dist
COPY --from=builder --chown=appuser:appuser /app/node_modules ./node_modules
USER appuser
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD node healthcheck.js
CMD ["node", "dist/index.js"]

Docker Compose for Development

# docker-compose.yml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DB_HOST=db
      - REDIS_HOST=redis
    depends_on:
      - db
      - redis

  db:
    image: postgres:15
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - db_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine

volumes:
  db_data:

Run entire stack: docker-compose up

How Architects Think About Docker

Image Size Matters

Base Image Size Pros Cons
ubuntu:latest 77MB Familiar Bloated with utils
alpine:latest 7MB Tiny Minimal tools
python:3.11-slim 125MB Balanced Just enough
python:3.11 1GB Full-featured Heavy

Benefits & Pros

  • ✅ Consistency: Same container everywhere (laptop to prod)
  • ✅ Isolation: Apps don't interfere with each other
  • ✅ Dependency management: All libraries included
  • ✅ Easy deployment: Push image, run container
  • ✅ Microservices-friendly: Each service in own container

Pitfalls & Cons

❌ Storing secrets in images

Docker images are just tar files. Anyone can extract your API keys.

⚠️ Oversized images

1GB image takes 5+ minutes to push/pull. Wastes time and bandwidth.

⚠️ Not cleaning up

Unused images pile up. Can consume 10s of GB.

Best Practices

1. Use Multi-Stage Builds

Build in one stage, copy only artifacts to final stage. Reduces image size dramatically.

2. Pin Base Image Versions

// Good
FROM python:3.11.1-slim

// Bad
FROM python:latest

3. Run as Non-Root

RUN useradd -m appuser
USER appuser

4. Use .dockerignore

node_modules
.git
.gitignore
.env
__pycache__
.pytest_cache

Real-World Example: Node.js App

# Dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci

FROM node:18-alpine
WORKDIR /app
RUN addgroup -g 1001 -S nodejs
RUN adduser -S nodejs -u 1001
COPY --from=builder /app/node_modules ./node_modules
COPY . .
USER nodejs
EXPOSE 3000
CMD ["node", "server.js"]

# Build and run
$ docker build -t myapp:1.0 .
$ docker run -p 3000:3000 myapp:1.0

Key Takeaway: Docker is standard for all modern applications. Master Dockerfile, Docker Compose, and image optimization. Don't store secrets in images.
